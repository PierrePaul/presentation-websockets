Modèle normal

  1. Le client fait un appel au serveur
  2. Le serveur répond dès que possible

Très pratique, simple, petite charge sur le serveur.

Lorsque l'environnement du serveur change, il doit attendre d'être appelé par le client pour lui donner l'information.

Les WebSockets

Les WebSockets permettent une connexion constante entre le client et le serveur. Donc lorsqu'un changement se passe sur le serveur, le client est mis à jour immédiatement.

Les deux écoles de pensée

Principalement, deux méthodologies sont utilisées pour les WebSockets :

  • Intégration complète au niveau du framework. Des services peuvent être programmés à même le framework, utilisé dans le language voulu. Beaucoup de librairies de qualité variable existe.
  • L'autre méthode très populaire, avoir un serveur de websocket (principalement fait en node.js) qui enregistre sur un système de "queue" comme Redis. Ensuite un framework peut lire la Queue Redis, traiter l'information et envoyer un message directement sur la Queue Redis pour être retraitée par le serveur de websocket. Le serveur de WebSocket va ré-envoyer l'information à tous les clients concernés.

Côté client

La connexion se fait en javascript, via l'objet WebSocket. L'objet expose 4 événements :

  • onopen : nous permet de savoir quand la connexion est établie.
  • onerror : nous permet de savoir quand une erreur se produit (par exemple si le serveur ne répond pas).
  • onmessage : lorsqu'un message est envoyé par le serveur de websocket.
  • onclose : lorsque le socket se ferme...

En pratique

connection = new WebSocket('ws://ws.pierre-paul.com:9001/echo');
connection.onopen = function () {};
connection.onerror = function (error) {};
connection.onmessage = function (e) {};    
connection.onclose = function (e) {}

On peut faire fonctionner pywebsocket en mode "standalone" ou en module apache. Le module standalone est particulièrement utile lors du développement.

En mode standalone, le fichier handler_map.txt permet de lier un fichier python à une URL.
